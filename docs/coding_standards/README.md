## Engineering Principles (Priority Order)

These principles are ordered by foundational importance. When they appear to conflict, the earlier
one takes precedence.

### 1. SOLID

The five object-oriented design principles. The most important for AthleteCanvas:

- **SRP** — one reason to change per class/module
- **OCP** — extend via new class, not by modifying existing ones (adapter registry pattern)
- **LSP** — any concrete adapter is a drop-in for BaseAdapter
- **ISP** — separate `IActivityReader`, `IActivityWriter`, `IEmbeddingStore` interfaces
- **DIP** — pipelines receive abstractions via constructor injection, never instantiate concrete deps

→ Full detail in [CODE_STYLE.md#solid-principles](CODE_STYLE.md)

### 2. DRY (Don't Repeat Yourself)

The same logic exists in exactly one place. Common extraction points: datetime normalisation,
dedup hashing, retry logic.

→ [CODE_STYLE.md#dry](CODE_STYLE.md)

### 3. Config over Code

No hardcoded operational values (paths, model names, thresholds, retry counts). All in
`backend/config/*.yaml`, loaded via typed Pydantic `AppConfig`.

→ [CODE_STYLE.md#config-over-code](CODE_STYLE.md)

### 4. Idempotence

Pipelines produce the same state whether run once or ten times. All writes are upserts.

→ [CODE_STYLE.md#idempotence](CODE_STYLE.md)

### 5. Fail-Fast at Startup

Config and path validation happens before the first request is accepted. Bad config = immediate
crash with a clear message, not a silent mid-run failure.

→ [CODE_STYLE.md#fail-fast-at-startup](CODE_STYLE.md)

### 6. Contract-Driven Development

All data flows between components via Pydantic DTOs. No raw `dict` passing between layers.

→ [CODE_STYLE.md#contract-driven-development](CODE_STYLE.md)

## Code Conventions Summary

| Convention | Rule |
|---|---|
| Type hints | Mandatory on all functions and methods, including tests |
| Datetime | Always UTC-aware; enforced by `field_validator` at DTO boundary |
| Imports | Three sections (stdlib / third-party / project), alphabetical, never inside functions |
| Line length | 100 characters maximum |
| File headers | `@layer`, `@dependencies`, `@responsibilities` — auto-generated by `scaffold_artifact` |
| Docstrings | Module: comprehensive. Class: one-line. Method: brief Google style |
| Pydantic examples | `json_schema_extra` with ≥ 2 examples on all exported DTOs |
| Global state | Forbidden — use constructor injection |
| Magic values | Forbidden — all in config YAML |
| Naive datetimes | Forbidden — UTC-aware only |

## Tooling

Quality gates are run via the **`run_quality_gates` MCP tool** — not manually. Use the manual
ruff/mypy commands in QUALITY_GATES.md only when debugging a specific gate failure.

Scaffolding is done via **`scaffold_artifact`** — not by creating files manually. Scaffolded files
auto-include correct headers, import structure, and Pydantic conventions.
